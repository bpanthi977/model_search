# Retrain Study

## Purpose

The `retrain-study/` directory contains an empirical study comparing two approaches for adapting
a surrogate model trained on one LULESH domain size to a different (larger) domain size:

- **Scratch**: Train a new model from random initialization on the new domain's data.
- **Retrain**: Start from a checkpoint trained on the original domain and continue training on
  the new domain's data (transfer learning / fine-tuning).

The main finding: **retraining for 10 epochs outperforms scratch training for 60 epochs** on the
new domain, while also maintaining accuracy on the original domain. See `docs/research_overview.md`
for a summary of results.

## Directory Structure

```
retrain-study/
├── 01 create_models.sh         # Export TorchScript model.pt from checkpoints
├── 02 compute_energy_mae.sh    # Run LULESH with each model, measure Energy MAE
├── 02 compute_energy_mae_all.sh# Run the above for all grid sizes (30, 40, 50, 60)
├── 03 collect_summary.sh       # Aggregate results into summary.csv
├── 04 create_plot.py           # Plot Energy MAE and val loss vs epochs
├── 05 scaling_plot.py          # Plot how MAE scales across domain sizes
├── energy_mae.sh               # Core script: runs one LULESH + computes Energy MAE
├── summary.csv                 # Aggregated results table (generated by step 03)
└── plots/                      # Generated figures (generated by steps 04, 05)
    ├── summary_grid_30-40.{png,pdf,svg}
    ├── summary_grid_30-50.{png,pdf,svg}
    ├── summary_grid_30-60.{png,pdf,svg}
    └── scaling_model_{2,3}.{png,pdf,svg}
```

## Models in the Study

The study uses two EVD models (Model #2: `3x24-200`, and Model #3: `512-512-512`) as base models,
each trained for different numbers of epochs and evaluated across four grid sizes (30, 40, 50, 60).

For each base model and each target grid size, two variants are compared:
- **Scratch (S)**: A model trained from zero on the new grid size's dataset.
- **Retrained (R)**: The base model's checkpoint, continued training on the new grid size's dataset.
- **Original (O)**: The base model itself (only valid for grid 30, the training domain).

The checkpoint directories are stored under `logs/` with their standard timestamped names. The
`continue_from` file inside a checkpoint directory marks it as a retrained run.

## Running the Study

### Step 1: Export TorchScript Models

```bash
cd retrain-study
./01\ create_models.sh
```

Runs `python main.py --create-model --checkpoint ...` for each relevant checkpoint to produce
`model.pt` files needed for LULESH inference.

### Step 2: Evaluate Across Domain Sizes

```bash
# Evaluate on one grid size (e.g. 40), using 4 parallel jobs
./02\ compute_energy_mae.sh 40 4

# Or run all grid sizes sequentially
./02\ compute_energy_mae_all.sh
```

For each model and each number of training epochs (sampled at intervals), this script:
1. Runs LULESH with the surrogate model via `SURROGATE_MODEL=... ./lulesh2.0 -s <grid_size>`.
2. Compares the resulting energy output to the baseline (no surrogate) and computes Energy MAE.
3. Saves results to `retrain-study/<grid_size>/<run_name>/e<epoch>-s<grid>/`:
   - `energy_mae.txt` — Energy MAE value
   - `execution_time.txt` — Wall-clock time
   - `Energy-*.bin` — Raw energy array from LULESH

The number of iterations is scaled with grid size (30→1500, 40→2000, 50→2500, 60→2500) to
keep simulation time comparable.

### Step 3: Collect Results

```bash
./03\ collect_summary.sh
```

Walks all experiment directories and writes `summary.csv` with columns:

| Column | Description |
|--------|-------------|
| `run` | Checkpoint directory name |
| `dataset` | Grid size the checkpoint was evaluated on |
| `is_retrain` | `1` if checkpoint has a `continue_from` file (i.e. was fine-tuned), `0` if trained from scratch |
| `epoch` | Number of training epochs for this checkpoint |
| `grid_size` | LULESH grid size used for evaluation |
| `mae` | Energy MAE |
| `val_loss` | Validation loss from `val_loss.csv` |
| `execution_time` | LULESH wall-clock time in seconds |

### Step 4: Generate Plots

```bash
python "04 create_plot.py"
python "05 scaling_plot.py"
```

`04 create_plot.py` generates one figure per target grid size (40, 50, 60), each with three
subplots:
- Energy MAE on grid 30 (original domain) vs. training epochs
- Energy MAE on grid N (new domain) vs. training epochs
- Validation loss vs. training epochs

Lines are labeled O (Original), S (Scratch), R (Retrained). This directly shows that the
retrained model reaches lower error faster than scratch training.

`05 scaling_plot.py` generates one figure per base model showing how Energy MAE changes
across domain sizes (40, 50, 60) for the best scratch and retrained variants at comparable
epochs. Shows the degradation curve with increasing domain size for each approach.

## How Retraining Works (in the Main Code)

Retraining uses the same `--checkpoint` resume mechanism as normal training. The key flag is that
the dataset is changed to one from the new domain size:

```bash
python main.py \
  --checkpoint logs/evd-study/20260218-105854-ysJr/ \
  --dataset.db_file dataset/EVD-grid40.h5 \
  --train.epoch 60    # 50 original + 10 new
```

Only `train.epoch`, `dataset.db_file`, and `study_name` may be overridden when resuming. The
model architecture and optimizer settings are locked to the original checkpoint.

When resumed, the original checkpoint directory is copied to a new timestamped directory. A
`continue_from` file is written with the name of the source checkpoint — this is how
`03 collect_summary.sh` detects retrained runs.

## Key Result

For EVD Model #3 (`512-512-512`, trained 50 epochs on grid 30):

- Retrained for **10 extra epochs** on grid 50 data → **better Energy MAE on grid 50** than a
  model trained **60 epochs from scratch** on grid 50 data.
- The retrained model also retained good accuracy on grid 30, while the scratch model had worse
  grid-30 accuracy.

This result held for grid sizes 40, 50, and 60. The benefit of retraining grows with domain size
since larger grids share more structure with the training domain than random initialization does.
